CONCEPT 01 What is a Go Interface?

An interface is a contract. It says: "I don't care what type you are — as long as you can do these things."

Real-world analogy: A USB port is an interface. Your laptop doesn't care if you plug in a mouse, keyboard, or flash drive — as long as it has the right connector shape (implements the contract), it works.
basic_interface.go 
// Define the interface — this is the CONTRACT
type Greeter interface {
    Greet() string
}

// Two DIFFERENT types, both satisfy the Greeter interface
type English struct{}
func (e English) Greet() string { return "Hello!" }

type Bangla struct{}
func (b Bangla) Greet() string { return "Assalamu Alaikum!" }

// This function accepts ANYTHING that satisfies Greeter
func SayHello(g Greeter) {
    fmt.Println(g.Greet())
}

func main() {
    SayHello(English{}) // prints: Hello!
    SayHello(Bangla{})  // prints: Assalamu Alaikum!
}
💡

The magic: In Go, you NEVER say "I implement this interface." If your type has all the required methods, it automatically satisfies the interface. This is called implicit implementation.

CONCEPT 02 Implicit vs Explicit

Unlike Java or C#, Go doesn't use implements keyword. The compiler figures it out automatically.

implicit.go 
// Java style (NOT Go):
// class Dog implements Animal { ... }  ❌

// Go style — just have the method, that's it!
type Animal interface {
    Sound() string
}

type Dog struct{ Name string }

// Dog satisfies Animal — no declaration needed ✅
func (d Dog) Sound() string {
    return "Woof"
}

MOTIVATION Why Do Interfaces Exist?

Without interfaces, you'd need a separate function for every type. With interfaces, one function handles all of them.

without_interface.go 
// ❌ WITHOUT interfaces — repetitive, fragile
func ProcessCreditCard(c CreditCard) { c.Charge(100) }
func ProcessPayPal(p PayPal)         { p.Charge(100) }
func ProcessCrypto(c Crypto)         { c.Charge(100) }
// Every new payment method = new function 😭

// ✅ WITH interfaces — one function rules them all
type PaymentMethod interface {
    Charge(amount float64) error
}

func ProcessPayment(p PaymentMethod, amount float64) {
    if err := p.Charge(amount); err != nil {
        fmt.Println("Payment failed:", err)
        return
    }
    fmt.Println("Payment successful!")
}
// New payment method? Just implement Charge() and done ✅

STEP 1 Define the Interface

Start by asking: "What do ALL payment methods need to do?" That becomes your interface.

payment.go 
package main

// PaymentMethod is the contract every payment type must fulfill
type PaymentMethod interface {
    Charge(amount float64) error
    Refund(amount float64) error
    Name() string
}
💡

Keep interfaces small. Go's standard library mostly uses 1-2 method interfaces. Small = more reusable.

STEP 2 Implement: Credit Card

creditcard.go 
package main

import "fmt"

type CreditCard struct {
    CardNumber string
    Balance    float64
}

func (c *CreditCard) Charge(amount float64) error {
    if c.Balance < amount {
        return fmt.Errorf("insufficient balance: have %.2f, need %.2f", c.Balance, amount)
    }
    c.Balance -= amount
    fmt.Printf("[CreditCard] Charged $%.2f. Remaining: $%.2f\n", amount, c.Balance)
    return nil
}

func (c *CreditCard) Refund(amount float64) error {
    c.Balance += amount
    fmt.Printf("[CreditCard] Refunded $%.2f\n", amount)
    return nil
}

func (c *CreditCard) Name() string {
    return fmt.Sprintf("CreditCard ****%s", c.CardNumber[len(c.CardNumber)-4:])
}

STEP 3 Implement: PayPal

paypal.go 
package main

import "fmt"

type PayPal struct {
    Email   string
    Balance float64
}

func (p *PayPal) Charge(amount float64) error {
    if p.Balance < amount {
        return fmt.Errorf("PayPal: not enough funds")
    }
    p.Balance -= amount
    fmt.Printf("[PayPal] Charged $%.2f from %s\n", amount, p.Email)
    return nil
}

func (p *PayPal) Refund(amount float64) error {
    p.Balance += amount
    fmt.Printf("[PayPal] Refunded $%.2f to %s\n", amount, p.Email)
    return nil
}

func (p *PayPal) Name() string {
    return "PayPal: " + p.Email
}

STEP 4 The Processor — Interface in Action

This is where interfaces shine. The processor doesn't care WHAT type it receives.

processor.go 
package main

import "fmt"

type Order struct {
    ID     string
    Amount float64
}

// ProcessOrder only knows about the PaymentMethod interface
// It has NO idea if it's CreditCard, PayPal, or Crypto
func ProcessOrder(order Order, payment PaymentMethod) {
    fmt.Printf("\n=== Processing Order %s ===\n", order.ID)
    fmt.Printf("Using: %s\n", payment.Name())

    if err := payment.Charge(order.Amount); err != nil {
        fmt.Printf("❌ Failed: %v\n", err)
        return
    }
    fmt.Println("✅ Order complete!")
}

// Process MULTIPLE payments — slice of interfaces!
func ProcessAll(orders []Order, methods []PaymentMethod) {
    for i, order := range orders {
        ProcessOrder(order, methods[i%len(methods)])
    }
}

STEP 5 Wire It Together in main.go

main.go 
package main

import "fmt"

func main() {
    // Create concrete types
    card := &CreditCard{CardNumber: "1234567890123456", Balance: 500}
    pp   := &PayPal{Email: "user@example.com", Balance: 200}

    // Store them as interfaces — this is polymorphism!
    var methods []PaymentMethod = []PaymentMethod{card, pp}

    orders := []Order{
        {ID: "ORD-001", Amount: 99.99},
        {ID: "ORD-002", Amount: 49.99},
        {ID: "ORD-003", Amount: 999.00}, // this will fail
    }

    ProcessAll(orders, methods)

    // Type assertion — check the real type at runtime
    for _, m := range methods {
        if cc, ok := m.(*CreditCard); ok {
            fmt.Printf("\nCredit card balance: $%.2f\n", cc.Balance)
        }
    }
}
🎯

Run it: go run . — You'll see each payment method processing orders differently, but the processor code never changed!

ADVANCED Type Assertions & Type Switches

Sometimes you need to get the real type back from an interface variable.

type_assertions.go 
// Type assertion — "I think this is a CreditCard"
var p PaymentMethod = &CreditCard{Balance: 100}

// Safe assertion (won't panic)
if cc, ok := p.(*CreditCard); ok {
    fmt.Println("It's a credit card with balance:", cc.Balance)
}

// Type switch — handle multiple types cleanly
func Describe(p PaymentMethod) {
    switch v := p.(type) {
    case *CreditCard:
        fmt.Printf("Credit card ending in %s\n", v.CardNumber[12:])
    case *PayPal:
        fmt.Printf("PayPal account: %s\n", v.Email)
    default:
        fmt.Printf("Unknown payment method: %T\n", v)
    }
}

ADVANCED Interface Composition

Interfaces can embed other interfaces — build bigger contracts from smaller ones.

composition.go 
// Small, focused interfaces
type Charger interface {
    Charge(amount float64) error
}

type Refunder interface {
    Refund(amount float64) error
}

// Composed interface — embeds both
type FullPayment interface {
    Charger
    Refunder
    Name() string
}

// Now you can accept just Charger for charge-only functions
func QuickCharge(c Charger, amount float64) error {
    return c.Charge(amount)
}

ADVANCED Testing with Mock Interfaces

This is where interfaces really pay off — you can inject fake implementations for testing.

payment_test.go 
// Mock implementation — only used in tests
type MockPayment struct {
    ShouldFail bool
    Charged    float64
}

func (m *MockPayment) Charge(amount float64) error {
    if m.ShouldFail {
        return fmt.Errorf("mock: charge failed")
    }
    m.Charged = amount
    return nil
}
func (m *MockPayment) Refund(float64) error { return nil }
func (m *MockPayment) Name() string        { return "MockPayment" }

func TestProcessOrder(t *testing.T) {
    mock := &MockPayment{}
    order := Order{ID: "TEST-1", Amount: 50.0}

    ProcessOrder(order, mock) // pass mock, not real payment

    if mock.Charged != 50.0 {
        t.Errorf("expected 50.0, got %v", mock.Charged)
    }
}

PITFALL 01 Nil Interface vs Nil Pointer

Common gotcha: A nil pointer inside an interface is NOT the same as a nil interface.
nil_trap.go 
var cc *CreditCard = nil
var p  PaymentMethod = cc

// p is NOT nil — it has type info (*CreditCard) but nil value!
fmt.Println(p == nil)  // false 😱

// Safe check: use a typed nil
var p2 PaymentMethod = nil
fmt.Println(p2 == nil) // true ✅

PITFALL 02 Pointer vs Value Receivers

receivers.go 
// If you define methods with pointer receivers...
func (c *CreditCard) Charge(amount float64) error { ... }

// You MUST use a pointer when assigning to interface
var p PaymentMethod = &CreditCard{}  // ✅ pointer
var p PaymentMethod = CreditCard{}   // ❌ compile error

// Rule: pointer receivers → use pointer. Value receivers → either works.

BEST PRACTICE Accept Interfaces, Return Structs

Go proverb: "Accept interfaces, return concrete types." Your functions should accept interfaces (flexible) but return concrete structs (clear).
best_practice.go 
// ✅ Accept interface (flexible input)
func ProcessPayment(p PaymentMethod) error { ... }

// ✅ Return concrete type (clear output)
func NewCreditCard(number string) *CreditCard {
    return &CreditCard{CardNumber: number, Balance: 0}
}

// ❌ Avoid returning interfaces unless necessary
func NewPayment() PaymentMethod { ... } // why?
